Silo · v0.1.0

A cache server
you run yourself.

Silo is a self-hosted key-value store, scoped into buckets. One bucket per app. One server for as many apps as you own.

Every value you set carries a time to live. Nothing is kept by accident. Every bucket has its own key, so one app's cache can never read another's.

01
API key
Used on admin routes only. Proves the caller owns this instance and can manage buckets.
02
Bucket key
Used on data routes only. Proves the caller is allowed to read and write to a specific bucket.
03
Your data
Stored with an expiry. Read back until it lapses, then gone.

Authentication

Silo uses two separate keys, each scoped to its own layer. They never overlap.

API key — admin routes

Issued once during silo init. Required on all /admin/* routes. Not required on data routes.

header 
Authorization: Bearer <api-key>

Bucket key — data routes

Issued when a bucket is created. Required on all /v1/* routes. Shown once — if lost, rotate it. Silo only stores a hash.

header 
Authorization: Bucket <bucket-key>
Key separation
Admin routes use Bearer with the API key. Data routes use Bucket with the bucket key. They are completely independent — a data request never needs the API key, and an admin request never needs a bucket key.

Time to live

Every value expires. If you don't set ttl on a write, the server's default applies — it's configured per instance, never "forever."

POST /admin/buckets/:bucketId/rotate api key
Issues a new bucket key and invalidates the old one immediately. Use this if a key leaks, or if you've lost it.
request 
curl -X POST https://your-host/admin/buckets/236ba04d-.../rotate \
  -H "Authorization: Bearer <api-key>"

// 200 ok
{
  "bucketId": "236ba04d-933f-4056-9591-eb20df52cca3",
  "bucketKey": "silo_bucket_a93dd0..."
}
Note
The previous key stops working the moment this returns. Update every app using this bucket before you rotate, not after.

GET /v1/:bucketId/:key bucket key
Reads a value back. Returns 404 if the key never existed or has expired.
request 
curl https://your-host/v1/236ba04d-.../poll-results \
  -H "Authorization: Bucket <bucket-key>"

// 200 ok
{
  "value": { "yes": 412, "no": 88 },
  "expiresAt": 1750000060000
}

// 404 not found
{ "error": "not found" }
DELETE /v1/:bucketId/:key bucket key
Removes a key before its expiry. Always returns success, whether or not the key existed.
request 
curl -X DELETE https://your-host/v1/236ba04d-.../poll-results \
  -H "Authorization: Bucket <bucket-key>"

// 200 ok
{
  "bucketId": "236ba04d-933f-4056-9591-eb20df52cca3",
  "key": "poll-results",
  "deleted": true
}

TypeScript SDK

The SDK is a single class that wraps the data routes. Zero dependencies — just fetch.

usage 
// import from the sdk
import { Silo } from "./sdk/mod.ts";

const cache = new Silo({
  url: "http://localhost:8000",
  bucketId: "236ba04d-933f-4056-9591-eb20df52cca3",
  bucketKey: "silo_bucket_8f2a1c...",
});

// set a value (ttl in seconds, optional)
await cache.set("poll-results", { yes: 412, no: 88 }, 60);

// get it back — returns null if expired or missing
const data = await cache.get("poll-results");

// delete before expiry
await cache.delete("poll-results");
Auth
The SDK uses the Bucket auth header automatically. You only need the bucket key — not the API key.

Errors

Errors are JSON, with a short message at minimum. Validation failures include a field-level breakdown.

StatusMeaning
400The request body didn't match the expected shape.
401The API key or bucket key was missing or wrong.
404The bucket or key doesn't exist, or has expired.
409Conflict — account already initialised, or a concurrent modification race.
